# Spec Review — evalwire vs SPECS.md

> Divergences between the current implementation and the original specification.
> Severity levels: **Critical** (breaks the contract), **Major** (meaningful
> functional gap), **Minor** (cosmetic or low-impact).

---

## 1. Phoenix API Surface — `DatasetUploader` (Critical)

**Spec (§8 and §14)** mandates the namespaced client API:

```python
client.datasets.create_dataset(dataframe, name, input_keys, output_keys)
client.datasets.get_dataset(name=tag)
client.datasets.delete_dataset(id=dataset.id)
client.datasets.add_examples(dataset_id=dataset.id, examples=[...])
```

**Implementation (`src/evalwire/uploader.py`)** uses a flat API:

```python
client.upload_dataset(dataset_name, dataframe, input_keys, output_keys)
client.get_dataset(name=name)
client.append_to_dataset(dataset_name, dataframe, input_keys, output_keys)
```

These are different method names with different signatures. If Phoenix 13.x
exposes the namespaced API (`client.datasets.*`), the implementation will raise
`AttributeError` at runtime. The test suite and mock fixtures also mirror the
flat API, so tests pass while the real API contract remains unverified.

---

## 2. Phoenix API Surface — `ExperimentRunner` (Critical)

**Spec (§9, §14, and §16)** specifies:

```python
experiment = client.experiments.run_experiment(
    dataset=dataset,
    task=task,
    evaluators=[...],
    experiment_name="...",
    experiment_metadata={...},
    dry_run=False,
)
```

Notably, §16 (Migration Guide) explicitly instructs:

> "Replacing `phoenix.experiments.run_experiment(...)` (module-level function)
> with `client.experiments.run_experiment(...)` (client method)."

**Implementation (`src/evalwire/runner.py:95-104`)** does the opposite — it
imports and calls the module-level function:

```python
from phoenix.experiments import run_experiment
result = run_experiment(dataset=dataset, task=task, ...)
```

This is the exact legacy pattern the spec says to move away from. It means
`self.client` is never used to dispatch experiment runs, and the client's
authentication/endpoint configuration may not be respected.

---

## 3. `on_exist="overwrite"` Logic is Incomplete (Critical)

**Spec (§8):**

> `"overwrite"`: retrieve the dataset by name with
> `client.datasets.get_dataset(name=tag)`, **delete it** via
> `client.datasets.delete_dataset(id=...)`, then re-create.

**Implementation (`src/evalwire/uploader.py:126-168`):**

The `on_exist="overwrite"` branch falls straight through to `upload_dataset`
without retrieving or deleting the existing dataset. There is no delete step at
all. Whether this results in an error, a silent overwrite, or duplicate data
depends entirely on what `upload_dataset` does internally — which is not part
of the public contract and cannot be relied upon.

---

## 4. `ExperimentRunner.concurrency` is Not Implemented (Major)

**Spec (§9):**

> `concurrency` — number of experiments to run in parallel (default: 1,
> sequential).

**Implementation (`src/evalwire/runner.py`):**

`self.concurrency` is stored in `__init__` but never referenced again. The
`run()` method always iterates sequentially over experiments with a plain
`for` loop. The `--concurrency` CLI option is accepted and validated, but has
no effect whatsoever.

---

## 5. `__init__.py` Auto-Generation Not Implemented (Major)

**Spec (§5):**

> `__init__.py` — optional; **auto-generated by evalwire if absent**

**Implementation (`src/evalwire/runner.py`):**

`_discover()` never writes an `__init__.py` file. It simply imports each
experiment module directly via `importlib.util.spec_from_file_location`. The
auto-generation behavior described in the spec is absent.

---

## 6. `ExperimentRunner` Does Not Call `client.datasets.get_dataset` (Major)

**Spec (§9 discovery logic, step 4):**

> Retrieve the Phoenix dataset whose name equals `<name>` via
> `client.datasets.get_dataset(name=name)`.

**Implementation (`src/evalwire/runner.py:188-194`):**

```python
def _get_dataset(self, name: str) -> Any | None:
    return self.client.get_dataset(name=name)
```

Consistent with finding #1, this uses the flat `client.get_dataset` instead of
the namespaced `client.datasets.get_dataset`. Same root cause, noted separately
because it affects the runner independently of the uploader.

---

## 7. `DatasetUploader` Constructor Default Signature (Minor)

**Spec (§8):**

```python
def __init__(
    self,
    input_keys: list[str] = ("user_query",),
    output_keys: list[str] = ("expected_output",),
    ...
)
```

**Implementation (`src/evalwire/uploader.py:37-53`):**

```python
def __init__(
    self,
    input_keys: list[str] | None = None,
    output_keys: list[str] | None = None,
    ...
)
```

The spec shows list/tuple defaults; the implementation uses `None` with
internal fallback logic. Functionally equivalent, but the public signature
differs: callers inspecting `help(DatasetUploader)` or IDE tooltips see
`None` instead of `["user_query"]`, which is less self-documenting.

## 8. `--dry-run` CLI Flag Semantics (Minor)

**Spec (§11):**

> `--dry-run [INTEGER]` — Run without uploading results. Optional count of
> examples.

The spec implies bare `--dry-run` maps to `True` (boolean) and
`--dry-run 5` maps to `5` (int).

**Implementation (`src/evalwire/cli.py:133-141`):**

```python
@click.option(
    "--dry-run",
    type=int,
    is_flag=False,
    flag_value=1,
    ...
)
```

Bare `--dry-run` passes `1` (int), not `True` (bool). This is functionally
equivalent in truthiness checks, but the `dry_run: bool | int` type contract
in `ExperimentRunner` is not honoured precisely, and downstream code checking
`dry_run is True` would behave differently.

---

## 9. `build_subgraph` Return Type Annotation (Minor)

**Spec (§6.1):**

```python
def build_subgraph(...) -> CompiledStateGraph: ...
```

**Implementation (`src/evalwire/langgraph.py:9-16`):**

```python
def build_subgraph(...) -> Any: ...
```

The return type is `Any` instead of `CompiledStateGraph`. Because
`langgraph` is an optional dependency the import must be guarded, but the
annotation could still use a string literal (`"CompiledStateGraph"`) or
`TYPE_CHECKING` guard to preserve the typed contract without a hard import.

---

## 10. `invoke_node` Not Used in Demo Task (Minor)

**Spec (§6.2)** introduces `invoke_node` as the idiomatic way to invoke a
single node directly, bypassing graph compilation. The demo
`demo/experiments/rag_pipeline/task.py` instead:

1. Calls `build_subgraph` (a full graph compilation step)
2. Manually constructs `RAGState` with a `HumanMessage`
3. Calls `await _subgraph.ainvoke(state)`

This is exactly the pattern `invoke_node` was designed to replace. Using
`invoke_node` would reduce the demo to:

```python
async def task(example) -> list[str]:
    result = await invoke_node(retrieve, example.input["user_query"], RAGState)
    return result["retrieved_titles"]
```

The demo is a primary usage example and should demonstrate the full API.
